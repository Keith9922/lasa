## 2026-05-04

### D-009 · Stori 残留代码移到 `_archive_stori/` 视觉屏蔽

- **结论**：把上一个项目（Stori 简历教练）的源码、配置、README、manifest 移到根目录下的 `_archive_stori/` 子目录，本工作区只看 `docs/` `research/` `prototype/` `public/` `src/`（待新建）
- **为什么**：worktree 从 main 拉出来时继承了 Stori 代码，看着乱。用户希望"屏蔽不看，但不删除"
- **排除**：
  - 直接 `git rm` 删除 — 用户偏好保守，先屏蔽
  - 加到 `.gitignore` — 文件已 tracked，gitignore 不生效
  - 用 `git update-index --skip-worktree` — 不直观，新人难理解
- **风险**：零。git 历史完全不变；如要恢复 Stori，从 main 拉新 worktree 或把文件 mv 回来即可

### D-008 · 把开发记录沉淀成 docs/

- **结论**：在 `docs/` 下建立一套开发档案：spec / interaction / animation / design-system / ai-integration / decisions / roadmap
- **为什么**：聊天上下文随时会丢；后续接力开发或回看决策需要可检索的离线记录
- **排除**：写在根 README — 那是项目自身的，混在一起难看

### D-007 · 便便颜色用写实色而非卡通色

- **结论**：放弃荧光绿/纯紫/亮橙，全部换成棕系闷调（健康棕 #6F4E37 / 绿褐 #5A5E2E / 暗红褐 #5C3025 等）
- **为什么**：用户反馈"真实便便不会这么红这么绿"。现实里所有颜色都是棕色被某种色素轻染，而非完全变色。写实色更可信
- **排除**：每种颜色单出 7 张图（成本×7） — 用 CSS filter 给单一基础色做色相偏移即可

### D-006 · Bristol 形象 SVG 起步，后期换 AI 生图

- **结论**：v0.1 用参数化 SVG（已实现 7 类）保证产品能跑；上线前用 GPT/Midjourney 生成 7 张高保真插画 + 透明底替换
- **为什么**：SVG 速度快、可参数化（颜色/油亮无需重出图）、零外部依赖；但"温度感"不如真插画。两步走兼顾速度和品质
- **排除**：从一开始就找插画师 — 周期太长，挡住其他开发

### D-005 · "马桶接屎 + 拍立得" 出卡动效

- **结论**：3.5 秒序列：屏幕震 → 马桶弹起 → 屎掉落（带挤压拉伸） → 水花涟漪 → 拍立得打印弹出 → 臭气线
- **为什么**：用户给的三个候选（出票机/拍立得/马桶接屎），合并后最有戏；马桶提供情境，拍立得是分享物，结合既沙雕又能输出可分享卡片
- **排除**：纯翻牌动画 — 太普通，没有产品特色

### D-004 · 双 Tab 输入（快捷选择 + 描述一下）

- **结论**：首屏两个 Tab 共享同一今日摄入，用户可任意切换混用
- **为什么**：兼顾"小白零门槛"（点食物芯片）和"重度玩家爽感"（自然语言爽快表达）。AI 解析填补两者之间的灵活度
- **排除**：单纯文本输入 — 移动端打字慢、首次用户没思路；单纯食物卡 — 食物库再大也覆盖不全

### D-003 · 不做登录、不做云存储

- **结论**：MVP 全部本地化；最多用 `localStorage` 存最近一次结果方便分享回看
- **为什么**：登录是巨大的体验摩擦点；这是娱乐工具不是工具型应用，用户来玩一次就走，没动力注册；云存储要后端 + 安全成本
- **排除**：注册账号 + 历史记录 — 后续如果数据证明用户有重复使用需求再加（先 localStorage 试水）

### D-002 · 对内/对外 AI 服务商命名严格分离

- **结论**：所有用户可见文案只说"AI 分析" / "AI 解析"，**绝不出现 MiniMax、GPT 等**；内部技术文档和代码注释可以正常提
- **为什么**：用户要求；同时降低品牌耦合（未来切换服务商无成本）
- **排除**：在 UI 上显示模型 Logo 增加信任感 — 反而显得依赖外部、不专业

### D-001 · 沙雕但不下三路

- **结论**：调性"适度沙雕"⭐⭐⭐⭐ / 5；玩梗、自嘲、网友体；忌粗口、人身攻击、暗示性内容
- **为什么**：题材本身够离谱了，文案再过头会丢失广泛传播性；可爱卡通的便便插画更友好
